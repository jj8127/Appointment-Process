# 정보보호 대학원 면접용 보안 조치 정리

최종 갱신: 2026-07-08

이 문서는 가람In(`fc-onboarding-app`)과 가람Link(`request_board`)에서 실제로 적용한 보안 조치 중 정보보호 대학원 면접에서 설명하기 좋은 내용을 정리한 문서입니다. 단순 기능 나열보다, 어떤 위협을 막기 위한 설계인지와 면접에서 어떻게 말하면 좋은지를 중심으로 구성했습니다.

운영 정책 기준: 주민번호는 업무 권한이 있는 trusted full-view 경로에서만 전체 값으로 표시합니다. 화면/응답에서 masked 또는 partial fallback을 사용하지 않으며, 복호화 실패 시 빈 값/null 또는 명시적 실패로 처리합니다. 로그, 테스트 출력, 문서 예시는 redaction 대상입니다.

## 1. 개인정보 보호: 주민번호 원문 최소화와 암호화

### 적용한 조치
- 주민번호 원문을 일반 프로필 테이블에 저장하지 않습니다.
- `fc_profiles` 같은 일반 프로필 영역에는 전체 주민번호를 저장하지 않고, 중복 확인이나 감사에 필요한 최소 파생값만 둡니다.
- 민감 식별정보는 별도 secure 영역에 암호화 저장합니다.
- 가람Link는 주민번호를 AES-256-GCM 방식으로 암호화하는 로직을 사용합니다.
- 화면 표시는 권한 있는 trusted full-view 조회가 성공한 경우에만 전체 주민번호를 사용합니다. masked/partial fallback은 운영 계약으로 사용하지 않습니다.
- 로그에는 주민번호 패턴과 `resident`, `ssn`, `customer_ssn` 같은 민감 key를 자동 redaction합니다.

### 보안적으로 중요한 점
- 개인정보를 기능상 필요한 범위로만 저장하는 데이터 최소화 원칙을 적용했습니다.
- 조회가 자주 필요한 일반 데이터와 강한 보호가 필요한 민감 데이터를 분리했습니다.
- AES-GCM을 사용해 기밀성과 무결성 검증을 함께 고려했습니다.
- 로그를 통한 2차 유출 위험까지 줄였습니다.

### 면접에서 말할 포인트
> 주민번호 같은 고위험 개인정보는 일반 프로필 영역에 전체 값으로 두지 않고, 별도 secure 저장소나 암호화 컬럼을 통해 관리했습니다. 업무상 필요한 화면에서는 trusted server path로 복호화에 성공한 경우에만 전체 값을 보여주고, 실패 시 masked 값으로 대체하지 않습니다. 대신 로그와 테스트 출력에는 민감정보가 남지 않도록 redaction 레이어를 두었습니다. 단순히 암호화만 한 것이 아니라 저장 위치, 조회 경로, 실패 처리, 로그 노출까지 위협 모델에 포함했습니다.

## 2. 인증 보안: 비밀번호 해시와 로그인 실패 잠금

### 적용한 조치
- 가람In은 전화번호 + 비밀번호 기반 커스텀 인증을 사용합니다.
- 가람Link는 Bearer JWT 기반 API 인증을 사용합니다.
- 비밀번호는 평문 저장하지 않고 단방향 해시로 저장합니다.
- 가람Link는 `bcrypt`를 사용해 비밀번호를 해시합니다.
- 로그인 실패 5회 시 10분 잠금 정책을 적용합니다.
- 비밀번호 재설정 코드는 평문이 아니라 해시로 저장합니다.
- 재설정 코드에는 만료 시간, 요청 쿨다운, 시도 제한을 적용합니다.

### 보안적으로 중요한 점
- DB가 유출되어도 비밀번호 평문이 노출되지 않도록 했습니다.
- brute-force 공격을 줄이기 위해 실패 횟수 기반 잠금을 적용했습니다.
- 비밀번호 재설정 흐름도 인증 흐름만큼 위험하므로, 토큰 저장/만료/시도 제한을 적용했습니다.

### 면접에서 말할 포인트
> 인증 보안에서는 비밀번호 저장 방식뿐 아니라 로그인 시도 제한과 재설정 흐름까지 함께 봤습니다. 비밀번호는 단방향 해시로 저장하고, 로그인 실패가 누적되면 일정 시간 잠금 처리했습니다. 재설정 코드도 평문 저장하지 않고 TTL과 시도 제한을 둬서 계정 탈취 가능성을 줄였습니다.

## 3. 서비스 간 인증: 가람In - 가람Link 브리지 토큰

### 적용한 조치
- 가람In 로그인 시 가람Link 연동용 `requestBoardBridgeToken`을 발급합니다.
- 가람Link는 `bridge-login`에서 bridge token을 검증합니다.
- bridge token은 HMAC SHA-256 서명을 사용합니다.
- 서명 비교는 timing attack을 줄이기 위해 `timingSafeEqual`을 사용합니다.
- bridge token에는 전화번호, 역할, 소속, 발급 시각, 만료 시각이 포함됩니다.
- bridge secret은 가람In과 가람Link 양쪽 환경변수로 관리합니다.
- 비밀번호 동기화도 별도 sync token으로 보호합니다.

### 보안적으로 중요한 점
- 서로 다른 두 서비스가 같은 사용자를 신뢰해야 하므로, 단순 파라미터 전달이 아니라 서명된 토큰을 사용했습니다.
- 토큰에 만료 시간을 포함해 탈취 후 재사용 위험을 줄였습니다.
- 역할 매핑을 제한해 관리자 권한이 의도치 않게 다른 서비스로 전파되지 않도록 했습니다.
- secret 회전은 양쪽 서비스가 함께 이뤄져야 하므로 운영 runbook 대상입니다.

### 면접에서 말할 포인트
> 가람In과 가람Link는 별도 서비스이기 때문에 서비스 간 신뢰 경계를 명확히 두었습니다. 사용자를 넘길 때 전화번호나 role을 그대로 믿지 않고, HMAC 서명된 bridge token을 발급하고 수신 서비스에서 검증합니다. 또한 토큰에 만료 시간을 넣고, role mapping을 제한해 권한 상승을 막았습니다.

## 4. 권한 분리: Role-Based Access Control

### 적용한 조치
- 가람In은 관리자, 본부장, FC 역할을 분리합니다.
- 본부장 계정은 읽기 전용 UI 정책을 적용합니다.
- 관리자 전용 API는 세션 쿠키와 역할을 함께 검증합니다.
- 가람Link API는 `requireRole` middleware로 접근 권한을 제한합니다.
- 가람Link 직접 런타임 역할은 기본적으로 `fc`, `designer` 중심으로 제한합니다.
- 일반 `admin`은 가람Link bridge의 직접 대상이 아닙니다.

### 보안적으로 중요한 점
- 인증된 사용자라도 모든 데이터와 기능에 접근할 수 없도록 역할별 권한을 분리했습니다.
- 프론트엔드 버튼 숨김에만 의존하지 않고 서버 API에서도 권한을 검증합니다.
- 서비스 간 role mapping을 제한해 권한 상승 가능성을 낮췄습니다.

### 면접에서 말할 포인트
> 권한 관리는 UI 제어만으로 끝내지 않고 서버에서 role을 검증하도록 했습니다. 특히 가람In의 관리자/본부장/FC와 가람Link의 FC/설계사 역할이 다르기 때문에, 서비스 간 role mapping을 명시적으로 제한했습니다. 인증과 인가를 분리해서 생각한 사례로 설명할 수 있습니다.

## 5. 데이터베이스 보안: RLS와 service-role 경로 분리

### 적용한 조치
- Supabase 주요 테이블에 RLS를 적용합니다.
- 가람In 게시판 관련 테이블과 스토리지는 service role 경로 중심으로 접근합니다.
- 가람Link는 `fc_company_codes`, `direct_conversations`, `direct_messages`, `direct_message_attachments` 등에 RLS를 적용합니다.
- 클라이언트가 service-role key를 직접 사용하지 않도록 서버 경로에서 처리합니다.
- DB schema 변경은 migration으로만 수행합니다.

### 보안적으로 중요한 점
- 클라이언트 탈취나 anon key 노출 상황에서도 DB 직접 접근 피해를 줄일 수 있습니다.
- service-role key는 강한 권한이므로 서버/Edge Function 내부로 격리했습니다.
- RLS와 서버 검증을 함께 사용해 방어 계층을 나눴습니다.

### 면접에서 말할 포인트
> Supabase 환경에서는 anon key가 클라이언트에 노출될 수 있다는 전제를 두고 RLS를 적용했습니다. 강한 권한이 필요한 작업은 service-role key로 처리하되, 이 key는 서버나 Edge Function 내부에서만 사용하도록 분리했습니다. 즉 클라이언트 권한, RLS, 서버 권한을 계층화했습니다.

## 6. 요청 위조와 API 보호: CSRF, Origin 검증, rate limit

### 적용한 조치
- 가람In 민감 서버 액션에서 Origin/Referer 검사를 수행합니다.
- 문서 변경, 위촉 상태 변경, 계정 삭제 등 일부 서버 액션에 rate limit을 적용합니다.
- 관리자 API는 세션과 역할을 함께 확인합니다.
- 가람Link는 JWT middleware와 role middleware로 API 접근을 제한합니다.
- CORS 허용 origin은 환경변수 기반으로 관리합니다.

### 보안적으로 중요한 점
- 로그인된 사용자의 브라우저를 악용하는 CSRF 위험을 줄였습니다.
- 민감 작업에 rate limit을 두어 자동화 공격과 실수 반복 호출을 줄였습니다.
- CORS 설정을 운영 환경변수로 관리해 허용 출처를 제한할 수 있게 했습니다.

### 면접에서 말할 포인트
> 웹 보안에서는 인증만으로 충분하지 않기 때문에 CSRF와 origin 검증도 적용했습니다. 특히 관리자 작업처럼 영향이 큰 API는 세션, role, origin을 함께 확인하고 일부 작업에는 rate limit을 적용했습니다. 공격자가 사용자의 로그인 상태를 악용하는 시나리오를 고려했습니다.

## 7. 첨부 파일 보안: 타입, 크기, signed URL

### 적용한 조치
- 파일 업로드는 서버 경유 구조를 사용합니다.
- 가람Link 첨부 파일은 4MB 제한을 둡니다.
- 허용 파일은 PDF와 이미지로 제한합니다.
- 허용 이미지 형식: jpg, jpeg, png, webp, gif, bmp, heic, heif
- 비공개 Storage bucket과 signed URL을 사용합니다.
- Storage object 권한과 DB metadata 권한을 함께 판단합니다.

### 보안적으로 중요한 점
- 임의 파일 업로드를 막기 위해 MIME allowlist와 크기 제한을 적용했습니다.
- 파일을 public URL로 직접 노출하지 않고 signed URL로 제한했습니다.
- object와 metadata 권한이 불일치할 수 있으므로 둘을 함께 검증 대상으로 봤습니다.

### 면접에서 말할 포인트
> 첨부 파일은 단순 저장 기능이 아니라 보안 위험이 큰 입력 경로로 봤습니다. 파일 타입과 크기를 제한하고, public bucket 대신 signed URL을 사용했습니다. 또한 Storage object만 보는 것이 아니라 DB metadata 권한까지 함께 봐야 접근 제어가 완성된다고 판단했습니다.

## 8. 알림과 푸시 보안: 서버 중심 fanout

### 적용한 조치
- 알림 저장과 push fanout의 기준 경로를 `fc-notify`로 통일했습니다.
- 게시판 글 생성은 `board-create`를 통해 `fc-notify` 계약을 유지합니다.
- Web Push VAPID key와 admin push secret은 환경변수로 관리합니다.
- Web Push identity row는 권한의 기준으로 사용하지 않습니다.
- 가람Link Web Push endpoint는 전역 unique로 관리합니다.

### 보안적으로 중요한 점
- 클라이언트가 임의로 알림을 만들거나 push 대상자를 조작하지 않도록 서버 경로로 통제했습니다.
- push token이나 endpoint는 식별 보조 정보일 뿐 권한 판단의 기준으로 삼지 않습니다.
- push secret은 운영 secret으로 별도 관리합니다.

### 면접에서 말할 포인트
> 알림과 push도 보안 관점에서는 신뢰 경계가 필요했습니다. 클라이언트가 직접 알림 row를 만들게 하지 않고, 서버의 표준 fanout 경로를 거치게 했습니다. 또한 push token은 권한의 근거가 아니라 발송 대상 식별자일 뿐이라는 점을 분리했습니다.

## 9. 운영 보안: secret 관리와 위험 옵션 통제

### 적용한 조치
- bridge secret, password sync token, VAPID key, admin push secret을 환경변수로 관리합니다.
- 하드코딩된 운영 credential script를 금지합니다.
- service-role key가 들어간 ops script는 운영 DB 대상 실행을 제한합니다.
- SMS bypass, test code, 테스트 SMS 모드는 운영 사용 금지 대상으로 문서화했습니다.
- bridge secret과 password sync token은 양쪽 서비스에서 함께 관리해야 합니다.

### 보안적으로 중요한 점
- 애플리케이션 보안은 코드뿐 아니라 secret 운영 방식에 의존합니다.
- 테스트 편의 기능이 운영에 남으면 인증 우회 취약점이 될 수 있습니다.
- 서비스 간 secret은 한쪽만 바꾸면 장애가 되고, 양쪽 모두 유출되면 인증 경계가 무너질 수 있습니다.

### 면접에서 말할 포인트
> 구현 보안뿐 아니라 운영 보안도 별도로 관리했습니다. bridge secret, sync token, push secret 같은 값은 환경변수로 관리하고 하드코딩을 금지했습니다. 특히 SMS bypass 같은 테스트 편의 기능은 운영에서는 인증 우회 취약점이 될 수 있어 명시적으로 비활성 원칙을 문서화했습니다.

## 10. 면접에서 강조하기 좋은 전체 구조

### 핵심 메시지
- 개인정보는 일반 저장소 원문 최소화, 암호화, trusted full-view 조회, 로그 redaction을 함께 적용했습니다.
- 인증은 비밀번호 해시, 로그인 잠금, 재설정 토큰 보호까지 포함했습니다.
- 서비스 간 연동은 HMAC 기반 bridge token으로 신뢰 경계를 분리했습니다.
- 권한은 UI가 아니라 서버 API, RLS, role middleware에서 검증했습니다.
- 파일과 push 같은 부가 기능도 별도 공격면으로 보고 제한했습니다.
- secret과 테스트 우회 옵션은 운영 보안의 핵심 위험으로 문서화했습니다.

### 1분 답변 예시
> 제가 만든 서비스에서는 보험 설계 관련 개인정보를 다루기 때문에 주민번호 같은 민감정보 보호를 가장 중요하게 봤습니다. 일반 테이블에는 전체 주민번호를 저장하지 않고, 업무상 전체 조회가 필요한 경우에는 trusted server path에서 암호화 데이터를 복호화해 성공한 경우에만 전체 값을 보여주도록 했습니다. 실패 시에는 masked 값으로 대체하지 않고 fail closed로 처리합니다. 인증 쪽에서는 비밀번호 해시, 로그인 실패 잠금, 재설정 토큰 TTL을 적용했고, 가람In과 가람Link처럼 두 서비스가 연동되는 구간은 HMAC 서명 기반 bridge token으로 보호했습니다. 또한 Supabase RLS와 서버 service-role 경로를 분리해 클라이언트가 직접 민감 데이터에 접근하지 못하도록 했고, 첨부 파일과 push 알림도 서버 중심으로 권한을 검증하도록 설계했습니다.

### 추가 질문을 받았을 때 답할 주제
- 왜 주민번호 원문을 일반 테이블에 두지 않았는지
- AES-GCM을 선택한 이유
- 해시와 암호화의 차이
- JWT 인증과 role-based authorization의 차이
- HMAC 서명 token이 단순 파라미터 전달보다 안전한 이유
- RLS와 service-role 분리의 장점
- 로그 마스킹이 필요한 이유
- 테스트 우회 기능이 운영 취약점이 되는 이유

## 11. 솔직하게 말할 수 있는 한계와 개선 방향

- Codex/PC 기반 자동화는 PC 전원, 네트워크, 로그인 상태, 로컬 env secret에 의존하므로 운영 서버 스케줄러보다 보안성과 안정성이 낮습니다.
- service-role key를 사용하는 ops script는 실행 주체와 실행 환경을 더 엄격히 제한할 필요가 있습니다.
- bridge secret 회전 절차는 문서화되어 있으나, 장기적으로는 자동화된 secret rotation과 감사 로그를 강화할 수 있습니다.
- 민감정보 전체 조회 trusted path는 계속 최소 범위로 유지해야 합니다.
- 향후에는 접근 로그, 이상 탐지, 관리자 작업 감사 로그, secret manager 연동을 강화하는 것이 좋습니다.

### 면접에서 한계 설명 방식
> 현재 구현은 개인정보 최소화, 암호화, RLS, 서버 권한 분리를 적용했지만 완성형 보안이라고 보지는 않습니다. 특히 운영 secret 관리, 감사 로그, 자동화 실행 환경은 계속 강화해야 할 영역입니다. 그래서 현재 조치와 남은 리스크를 함께 문서화하고 있습니다.
